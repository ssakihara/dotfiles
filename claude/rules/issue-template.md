# 課題作成ルール（Backlog / GitHub Issue）

Backlog 課題・GitHub Issue を作成する際は、以下のテンプレートを必ず使用すること。

## 対象

- Backlog 課題の起票（`mcp__backlog__backlog_add_issue` 等）
- GitHub Issue の作成（`gh issue create` 等）

## テンプレート

```markdown
## 背景・目的
- 現状の課題：
- 期待する効果：

## 完了の定義(DoD)
- [ ]
- [ ]

## 対応内容
-
-

## 対応範囲外
-

## 関連情報
- 仕様書／デザイン：
- 関連課題：

## 補足
- 期限：
- 懸念点：
```

## 記載ガイドライン

| セクション | 記載内容 |
|---|---|
| **背景・目的** | なぜこの課題に取り組むのか。現状と理想のギャップを明記する |
| **完了の定義(DoD)** | チェックリスト形式で、何が満たされれば完了とみなすかを具体的に記述（Definition of Done） |
| **対応内容** | 実施する作業の概要。実装方針・変更予定の箇所など |
| **対応範囲外** | 今回の課題で扱わないこと。誤解を防ぐために明示する |
| **関連情報** | 仕様書・デザインファイル・関連課題へのリンク |
| **補足** | 期限・懸念点・前提条件など、上記に収まらない情報 |

## 記載時の注意

- 空のセクションは削除せず、該当なしの場合は `- なし` と明記する
- 完了の定義(DoD)は**検証可能な粒度**で書く（例: 「画面が表示される」ではなく「/users ページで一覧が表示される」）
- 関連課題は Backlog なら課題キー（`XXX-123`）、GitHub なら `#番号` で記述する
- 期限が未定の場合でも、目安（「今スプリント内」等）を記載する
