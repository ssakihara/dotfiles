# コードレビュールール

`code-reviewer` エージェント（または同等のレビュー作業）でコードレビューを行う際は、本ルールに従うこと。
レビューの起動タイミング・委譲ルールは `agents.md` を参照する。

## 対象範囲（IMPORTANT）

- **git diff ベースで対象を限定する。** 全ファイルではなく、変更分のみをレビューする。
  - PR レビュー: `git diff <base>...HEAD` の差分
  - 作業中レビュー: `git diff` / `git diff --staged` の差分
  - 直近コミットの確認: `git diff HEAD~1`
- 差分の周辺コード（呼び出し元・型定義・関連テスト）は文脈として読むが、**指摘は差分行に対してのみ行う**こと（差分外への指摘はノイズになる）。
- 差分が空の場合は「レビュー対象なし」と明示して終了する。

## レビュー観点（必須）

以下4観点はすべての変更に対して必ず確認する。観点ごとに該当する `rules/*.md` を併せて参照すること。

### 1. セキュリティ（→ `security.md`）

- ハードコードされたシークレット・APIキー・トークン
- 入力検証の欠落（型・範囲・形式）
- SQL インジェクション（パラメータ化クエリの徹底）
- XSS（HTMLサニタイズ、`v-html` / `dangerouslySetInnerHTML` の不適切利用）
- CSRF 対策の有無
- 認証・認可の境界（権限チェック漏れ、IDOR）
- エラーメッセージへの機密情報混入
- 依存パッケージの安全性（既知脆弱性・サプライチェーン）

### 2. パフォーマンス（→ `performance.md`）

- N+1 クエリ・ループ内クエリ
- 不要な `SELECT *`・必要カラム以外の取得
- インデックスが効かない WHERE / JOIN / ORDER BY
- バルク処理すべき箇所での逐次処理
- キャッシュの有無・TTL/無効化戦略の妥当性
- 並列化可能な await の直列実行（`Promise.all` 漏れ）
- ストリーム処理すべき大容量データのオンメモリ展開
- 外部APIコールのタイムアウト未設定

### 3. 可読性・保守性・命名

- 関数・変数・型の命名が意図を表しているか（略語より完全な単語）
- 1関数1責務になっているか（責務の混在・長すぎる関数）
- 過剰な抽象化・早すぎる一般化（YAGNI 違反）
- デッドコード・コメントアウトされた残骸
- 「何をしているか」を説明するだけのコメントは削除を提案
- 「なぜそうしているか」が非自明な箇所にコメントがあるか
- マジックナンバー・マジック文字列の定数化
- SQL は `sql-format.md` に従っているか

### 4. 型安全性・エラーハンドリング

- `any` / `unknown` のキャストでの誤魔化し
- null / undefined の取り扱い（オプショナルチェーン・デフォルト値）
- 例外境界の明示（どこで catch し、どこに伝搬させるか）
- 失敗しうる処理の戻り値設計（Result 型 / throw の使い分け）
- 外部入力の型ガード・スキーマ検証（zod 等）
- `Promise` の握りつぶし（未 await・未 catch）
- TypeScript の `strict` 設定下で警告が出ていないか

## 重大度ラベル（必須）

すべての指摘に重大度を必ず付与する。**4段階のいずれか**を使用すること。

| ラベル | 意味 | 対応方針 |
|---|---|---|
| **Critical** | セキュリティ事故・データ破壊・本番障害に直結する | **作業を停止し、ユーザーに即座に報告。マージ不可** |
| **High** | バグ・パフォーマンス劣化・重大な保守性問題。リリース前修正必須 | マージ前に必ず修正 |
| **Medium** | 改善が望ましい設計・可読性問題。次回以降でも可 | 修正推奨。時間があれば対応 |
| **Low** | 好み・軽微なスタイル・nit | 任意。無視してよい |

判定基準:
- セキュリティ問題は原則 **Critical** または **High**
- 漏洩シークレット・SQLi・認可バイパス → **Critical**
- N+1・キャッシュ欠落で実害が見込まれる → **High**
- 命名・コメントなど見た目の問題 → **Medium** 以下
- 個人の好みに留まるもの → **Low**

## 出力フォーマット（必須）

### 1. ファイルパス:行番号を必ず付与

すべての指摘は **`相対パス:行番号` 形式**で対象箇所を明示する。範囲指摘は `path:start-end` で記述する。

```
src/api/users.ts:42
src/composables/useAuth.ts:15-28
```

行番号がない指摘（「全体的に〜」等）は原則禁止。やむを得ない場合は `path:` のみ付与し、対象範囲を散文で説明する。

### 2. 修正提案コードを before / after で提示

指摘には可能な限り具体的な修正例を **before / after の diff 形式**で添える。
適用するだけでマージ可能なレベルを目指す。

````markdown
### [High] `src/api/users.ts:42` — N+1 クエリ

ループ内で都度クエリを発行している。`IN` 句で一括取得すべき。

before:
```ts
for (const id of ids) {
  const user = await db.query(`SELECT * FROM users WHERE id = $1`, [id])
  results.push(user)
}
```

after:
```ts
const results = await db.query(
  `SELECT
    id,
    name,
    email
  FROM users
  WHERE id = ANY($1)`,
  [ids],
)
```
````

修正提案が困難な場合（設計レベルの再考が必要等）は、その旨と検討すべき選択肢を明記する。

### 3. 全体構成

レビュー結果は以下の順で出力する:

1. **サマリ**: 重大度別の件数（例: Critical 1 / High 2 / Medium 3 / Low 0）と総評
2. **Critical**: 該当する指摘をすべて
3. **High**: 同上
4. **Medium**: 同上
5. **Low**: 同上（多すぎる場合は要点のみ）
6. **レビュー範囲**: どの diff を対象としたか（コミットハッシュ・ファイル数等）

## 実行ポリシー

### 起動タイミング

- コード変更後は **必ず** code-reviewer エージェントを起動する（`agents.md` 参照）
- ユーザーから明示的に依頼があった場合
- PR 作成前のセルフチェック

### Critical 検出時の挙動（IMPORTANT）

Critical を1件でも検出した場合、`security.md` の手順に従う:

1. 作業を停止する
2. 該当箇所と影響範囲をユーザーに即座に報告する
3. 漏洩シークレットがあれば**ローテーション手順を必ず案内**する
4. ユーザーの判断を仰いだうえで修正に着手する

Critical を黙ってスキップしたり、独断で修正してマージへ進めないこと。

### レビュー後の記録

再利用価値のある知見（プロジェクト固有のハマりポイント・設計判断）を得た場合、
`obsidian-knowledge.md` のルールに従って Obsidian Vault に記録する。
