# コードレビュールール

`code-reviewer` エージェント（または同等のレビュー作業）でコードレビューを行う際は、本ルールに従うこと。
レビューの起動タイミング・委譲ルールは `agents.md` を参照する。

レビュー結果は常に **GitHub PR コメント形式**（ラベル + 1〜2行の簡潔な指摘）で出力する。
デフォルトの出力先はチャットで、テキストラベル（`[must]` 等）を使う。

GitHub への投稿は例外であり、ユーザーから求められた場合のみバッジ画像付きの下書きコメントとして投稿する。

## 対象範囲（IMPORTANT）

- **git diff ベースで対象を限定する。** 全ファイルではなく、変更分のみをレビューする。
  - PR レビュー: `git diff <base>...HEAD` の差分
  - 作業中レビュー: `git diff` / `git diff --staged` の差分
  - 直近コミットの確認: `git diff HEAD~1`
- 差分の周辺コード（呼び出し元・型定義・関連テスト）は文脈として読むが、**指摘は差分行に対してのみ行う**こと（差分外への指摘はノイズになる）。
- 差分が空の場合は「レビュー対象なし」と明示して終了する。

## レビュー観点（必須）

以下4観点はすべての変更に対して必ず確認する。観点ごとに該当する `rules/*.md` を併せて参照すること。

### 1. セキュリティ（→ `security.md`）

- ハードコードされたシークレット・APIキー・トークン
- 入力検証の欠落（型・範囲・形式）
- SQL インジェクション（パラメータ化クエリの徹底）
- XSS（HTMLサニタイズ、`v-html` / `dangerouslySetInnerHTML` の不適切利用）
- CSRF 対策の有無
- 認証・認可の境界（権限チェック漏れ、IDOR）
- エラーメッセージへの機密情報混入
- 依存パッケージの安全性（既知脆弱性・サプライチェーン）

### 2. パフォーマンス（→ `performance.md`）

- N+1 クエリ・ループ内クエリ
- 不要な `SELECT *`・必要カラム以外の取得
- インデックスが効かない WHERE / JOIN / ORDER BY
- バルク処理すべき箇所での逐次処理
- キャッシュの有無・TTL/無効化戦略の妥当性
- 並列化可能な await の直列実行（`Promise.all` 漏れ）
- ストリーム処理すべき大容量データのオンメモリ展開
- 外部APIコールのタイムアウト未設定

### 3. 可読性・保守性・命名

- 関数・変数・型の命名が意図を表しているか（略語より完全な単語）
- 1関数1責務になっているか（責務の混在・長すぎる関数）
- 過剰な抽象化・早すぎる一般化（YAGNI 違反）
- デッドコード・コメントアウトされた残骸
- 「何をしているか」を説明するだけのコメントは削除を提案
- 「なぜそうしているか」が非自明な箇所にコメントがあるか
- マジックナンバー・マジック文字列の定数化
- SQL は `sql-format.md` に従っているか

### 4. 型安全性・エラーハンドリング

- `any` / `unknown` のキャストでの誤魔化し
- null / undefined の取り扱い（オプショナルチェーン・デフォルト値）
- 例外境界の明示（どこで catch し、どこに伝搬させるか）
- 失敗しうる処理の戻り値設計（Result 型 / throw の使い分け）
- 外部入力の型ガード・スキーマ検証（zod 等）
- `Promise` の握りつぶし（未 await・未 catch）
- TypeScript の `strict` 設定下で警告が出ていないか

## 指摘ラベル（必須）

すべての指摘に以下4種のいずれかのラベルを必ず付与する。

| ラベル | 意味 | 対応方針 | 文末トーン |
|---|---|---|---|
| **must** | 修正必須。バグ・セキュリティ・実害あるパフォーマンス問題 | マージ前に必ず修正 | 「[修正箇所]を[修正内容]のように修正してほしいです。」 |
| **imo** | 改善提案。設計・可読性・個人の意見 | 検討推奨。対応は任意 | 「[修正箇所]を[修正内容]のように修正したほうがいいと思います。」 |
| **fyi** | 参考情報・補足。対応不要 | 読むだけでよい | 説明調 |
| **q** | 質問・確認したいこと | 回答が必要 | 「[質問内容]ですか？」 |

判定基準:

- 漏洩シークレット・SQLi・認可バイパスなどのセキュリティ問題 → **must**（加えて「セキュリティ重大問題検出時の挙動」に従う）
- N+1・キャッシュ欠落で実害が見込まれる → **must**
- 命名・コメント・設計の好みに関する指摘 → **imo**
- 意図が読み取れず判断できない箇所 → **q**
- 修正不要だが知っておくと役立つ情報 → **fyi**
- 迷ったら弱い方に倒す（must か imo で迷ったら imo）。ただし**セキュリティ問題は例外なく must**

## 出力フォーマット（必須）

チャット・GitHub のどちらに出す場合も、1指摘 = 1コメントの形式で書く。

### コメントの書式

1. **1行目にラベルを置く。** チャットでは `[must]` 等のテキスト、GitHub ではバッジ画像（後述）。
2. **対象箇所を `相対パス:行番号` で明示する。** 範囲指摘は `path:start-end`。
3. **本文は3〜5行で簡潔に書く。** 「何が問題か」のみを書く。修正コード例や長い仕様引用は載せない。
4. **句読点で必ず改行する。** 「。」の後ろで改行し、1行1情報を徹底する。
5. **文末トーンをラベルに合わせる**（上記ラベル表を参照）。

簡潔で読みやすい文にするため、以下を心がける:

- **主語と述語を必ず書く**: 「〜は」「〜が」と文末の述語をセットにし、誰が何をしたかを明確にする。

### チャット出力の全体構成

1. **サマリ**: ラベル別の件数（例: must 1 / imo 2 / q 1 / fyi 0）と総評
2. **指摘リスト**: must → imo → q → fyi の順に、コメントの書式で列挙
3. **レビュー範囲**: どの diff を対象としたか（コミットハッシュ・ファイル数等）

出力例:

```markdown
## サマリ
must 1 / imo 1 / q 0 / fyi 0
タイムアウト未設定が1件。それ以外は問題なし。

## 指摘

[must] `server/utils/oidc.ts:42`
外部 API 呼び出し全般を `timeout` 付きに修正してほしいです。
IdP がハングするとリクエストが無期限にブロックされるためです。

[imo] `server/utils/oidc.ts:15`
`res` を `tokenResponse` のような完全な単語の名前に修正したほうがいいと思います。
呼び出し側で何のレスポンスか分からなくなるためです。

## レビュー範囲
git diff main...HEAD（3ファイル、abc1234）
```

### 詳細説明を求められた場合

ユーザーから特定の指摘について詳細を求められたときのみ、以下を展開して説明する:

- 問題のメカニズム（なぜ起きるか・どんな実害があるか）
- 修正提案コードを **before / after 形式**で提示（適用するだけでマージ可能なレベルを目指す）
- 修正提案が困難な場合（設計レベルの再考が必要等）は、その旨と検討すべき選択肢

最初の出力に先回りして詳細を載せないこと。簡潔さを優先する。

## GitHub への投稿（ユーザーが求めた場合のみ）

ユーザーから「GitHub にコメントして」と求められた場合のみ、PR に下書きコメントとして投稿する。
求められていないのに投稿してはならない。

### 投稿方法

1. **インラインコメントにする。** PR 全体へのコメントではなく、指摘対象の diff 行に直接紐付ける。
2. **下書き（pending review）のまま残す。** submit せず、ユーザーが内容を確認してから提出する。

### バッジ

GitHub ではテキストラベルの代わりに、チームの saved reply バッジをコメントの1行目に置く:

| ラベル | バッジ |
|---|---|
| must | `![must-badge](https://img.shields.io/badge/review-must-red)` |
| imo | `![imo-badge](https://img.shields.io/badge/review-imo-orange)` |
| fyi | `![fyi](https://img.shields.io/badge/review-fyi-orange.svg)` |
| q | `![q](https://img.shields.io/badge/review-q-success.svg)` |

`相対パス:行番号` は本文に書かない（インラインコメントは GitHub 上で対象行に表示されるため不要）。

投稿例:

```markdown
![must-badge](https://img.shields.io/badge/review-must-red)
外部 API 呼び出し全般を `timeout` 付きに修正してほしいです。
IdP がハングするとリクエストが無期限にブロックされるためです。
```

### API 手順・注意点

- `POST /repos/{owner}/{repo}/pulls/{number}/reviews` で `event` を省略すると PENDING（下書き）レビューになる。
- `comments[]` の `line` / `side` 指定が `position`（diff 内位置）として解釈されるケースがある。`GET /pulls/{number}/files` レスポンス各ファイルの `patch` フィールドから position を計算して検証するのが確実。
- pending review は1ユーザーにつき1件まで。修正は DELETE → 再作成が簡単。
- ローカルの行番号はリファクタ等で PR の diff とズレることがある。コメント投稿前に必ず `GET /pulls/{number}/files` の `patch` で行位置を確認する。

## 実行ポリシー

### 起動タイミング

- コード変更後は **必ず** code-reviewer エージェントを起動する（`agents.md` 参照）
- ユーザーから明示的に依頼があった場合
- PR 作成前のセルフチェック

### セキュリティ重大問題検出時の挙動（IMPORTANT）

漏洩シークレット・SQLi・認可バイパスなど、セキュリティ事故・データ破壊・本番障害に直結する問題を1件でも検出した場合、`security.md` の手順に従う:

1. 作業を停止する
2. 該当箇所と影響範囲をユーザーに即座に報告する
3. 漏洩シークレットがあれば**ローテーション手順を必ず案内**する
4. ユーザーの判断を仰いだうえで修正に着手する

黙ってスキップしたり、独断で修正してマージへ進めないこと。
ユーザーから GitHub への投稿を求められていた場合でも、must の下書きコメントを作るだけで済ませず、必ずチャットでユーザーに即座に報告する。

### レビュー後の記録

再利用価値のある知見（プロジェクト固有のハマりポイント・設計判断）を得た場合、
`obsidian-knowledge.md` のルールに従って Obsidian Vault に記録する。
