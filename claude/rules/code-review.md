# コードレビュールール

コードレビューの実行時は `/code-review` スキルに従うこと。
本ルールはスキルに委譲できない「常時適用のポリシー」のみを定義する。

## 対象範囲（IMPORTANT）

- **git diff ベースで対象を限定する。** 全ファイルではなく、変更分のみをレビューする。
  - PR レビュー: `git diff <base>...HEAD` の差分
  - 作業中レビュー: `git diff` / `git diff --staged` の差分
  - 直近コミットの確認: `git diff HEAD~1`
- 差分の周辺コード（呼び出し元・型定義・関連テスト）は文脈として読むが、**指摘は差分行に対してのみ行う**こと。
- 差分が空の場合は「レビュー対象なし」と明示して終了する。

## 指摘ラベル（必須）

すべての指摘に以下5種のいずれかのラベルを必ず付与する。

| ラベル | 意味 | 対応方針 |
|---|---|---|
| **must** | 修正必須。バグ・セキュリティ・実害あるパフォーマンス問題 | マージ前に必ず修正 |
| **nit** | 軽微な指摘。スタイル・命名の些細な改善 | 対応は任意。ブロッカーではない |
| **imo** | 改善提案。設計・可読性・個人の意見 | 検討推奨。対応は任意 |
| **fyi** | 参考情報・補足 | 読むだけでよい |
| **q** | 質問・確認したいこと | 回答が必要 |

判定に迷ったら弱い方に倒す（must か imo で迷ったら imo）。
ただし**セキュリティ問題は例外なく must**。

## GitHub への投稿（ユーザーが求めた場合のみ）

- 求められていないのに投稿してはならない。
- インラインコメントにし、**下書き（pending review）のまま残す**。submit しない。
- バッジ画像をコメントの1行目に置く:

## セキュリティ重大問題検出時（IMPORTANT）

漏洩シークレット・SQLi・認可バイパスなど、セキュリティ事故・データ破壊・本番障害に直結する問題を1件でも検出した場合:

1. 作業を停止する
2. 該当箇所と影響範囲をユーザーに即座に報告する
3. 漏洩シークレットがあれば**ローテーション手順を必ず案内**する
4. ユーザーの判断を仰いだうえで修正に着手する

黙ってスキップしたり、独断で修正してマージへ進めないこと。

## 実行ポリシー

### 起動タイミング

rules/agents.md の「コードレビュー（必須）」に従う。

### レビュー後の記録

再利用価値のある知見（プロジェクト固有のハマりポイント・設計判断）を得た場合、
`obsidian-knowledge.md` のルールに従って Obsidian Vault に記録する。
