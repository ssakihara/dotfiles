# コーディング規約

## シェルコマンド

- ファイル内容の検索には grep ではなく rg (ripgrep) を使うこと

## コメント

- **JSDoc は対象を絞る。** 公開API・引数や戻り値の意図が非自明な関数・複雑な処理を行う関数のみに付ける。自明な関数には書かない
- **見てわかる実装内容（What）はコメントに書かない。** 名前で表現する
- コメントは「なぜそうしているか（Why）」が非自明な箇所にのみ書く
- **コメントは最大3行まで。** 1つのコメントブロックが3行を超える場合は、不要な情報を削るか書き直すこと
- **修正の経緯・試行錯誤の過程をコメントに残さない。** 「最初は X だったが Y に変更した」「以前は～していたが～」のような履歴は git log / PR で追えるため不要
- 「なぜこの実装か」の理由は1〜2行で要点だけ書く。背景の長文説明は避ける

## テストケース

- テストケースは `it` ではなく `test` を使って記述すること

## TypeScript

- 変数宣言は必ず `const` を用いること。`let` は使ってはならない
- `let` を使わないとどうしても実現できない場合のみ、ユーザーに確認を行ってから利用すること

## GitHub CLI (`gh`)

`gh api` ではなく `gh pr view --json` 等のサブコマンドを優先すること。
`gh api` はサブコマンドでは取得できないデータ（インラインレビューコメント等）にのみ使用する。

## 検証

作業完了時、テスト実行・スクリーンショット・検証コマンドなど検証方法を必ず提供すること。
