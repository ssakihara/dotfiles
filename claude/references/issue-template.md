# 課題作成テンプレート — 詳細ガイド

## テンプレート

```markdown
## 背景・目的
- 現状の課題：
- 期待する効果：

## 完了の定義(DoD)
- [ ]
- [ ]

## 対応内容
-
-

## 対応範囲外
-

## 関連情報
- 仕様書／デザイン：
- 関連課題：

## 補足
- 期限：
- 懸念点：

## 用語集
<!-- GitHub: 脚注形式 / Backlog: 「- **用語**: 説明」のリスト形式。該当用語がなければ「- なし」 -->
[^yougo1]: 用語1の説明
[^yougo2]: 用語2の説明
```

## 記載ガイドライン

| セクション | 記載内容 |
|---|---|
| **背景・目的** | なぜこの課題に取り組むのか。現状と理想のギャップを明記する |
| **完了の定義(DoD)** | チェックリスト形式で、何が満たされれば完了とみなすかを具体的に記述（Definition of Done） |
| **対応内容** | 実施する作業の概要。実装方針・変更予定の箇所など |
| **対応範囲外** | 今回の課題で扱わないこと。誤解を防ぐために明示する |
| **関連情報** | 仕様書・デザインファイル・関連課題へのリンク |
| **補足** | 期限・懸念点・前提条件など、上記に収まらない情報 |
| **用語集** | 本文中に登場する一般的ではない用語を定義する（GitHub: 脚注形式 / Backlog: リスト形式） |

## 記載時の注意

- 空のセクションは削除せず、該当なしの場合は `- なし` と明記する
- 完了の定義(DoD)は**検証可能な粒度**で書く（例: 「画面が表示される」ではなく「/users ページで一覧が表示される」）
- 関連課題は Backlog なら課題キー（`XXX-123`）、GitHub なら `#番号` で記述する

### 期限の記載（IMPORTANT）

- 期限は**カレンダー上で特定できる日時・時期**で記載する
  - ✅ `6/20まで`、`6月中旬まで`、`2026/7/1 リリースまで`（「中旬」「月末」など幅のある表現も可）
  - ❌ `〇〇の作業が終わるまで`、`リファクタリングが落ち着いたら`、`なるべく早く`（人によって認識が変動するため禁止。**他の作業・イベントの完了を基準にした相対的な表現は不可**）
- 期限が未定の場合でも、目安となる日時（「6月中」「今月末まで」等）を必ず記載する

### 用語集の記載（IMPORTANT）

- 本文中に**一般的ではない用語**（プロジェクト固有の用語・社内用語・略語・専門用語など）がある場合、Markdown の脚注（footnote）[^footnote] を使って用語集を**必ず**作成する
- 本文中の用語に `[^ラベル]` で脚注参照を付け、`## 用語集` セクションに `[^ラベル]: 説明` の形式で定義を記載する
- ラベルは**英数字・ハイフン・アンダースコアのみ**を使用する（日本語ラベルはレンダリングされない場合がある）

  ```markdown
  決済処理は SBPS[^sbps] のリンク型[^link-type]を利用する。

  ## 用語集
  [^sbps]: SBペイメントサービス。ソフトバンクグループの決済代行サービス
  [^link-type]: 決済画面を SBPS 側でホスティングする接続方式。自社実装は API 型と呼ぶ
  ```

- **Backlog は脚注記法をサポートしない**ため、Backlog 課題では脚注の代わりに `## 用語集` セクションへ `- **用語**: 説明` のリスト形式で記載する

  ```markdown
  ## 用語集
  - **SBPS**: SBペイメントサービス。ソフトバンクグループの決済代行サービス
  - **リンク型**: 決済画面を SBPS 側でホスティングする接続方式。自社実装は API 型と呼ぶ
  ```

- 一般的ではない用語が本文に存在しない場合は、用語集セクションに `- なし` と明記する

[^footnote]: `[^ラベル]` で本文中に参照を置き、`[^ラベル]: 説明` で定義する Markdown 記法。GitHub Issue ではレンダリングされるが、Backlog では未対応
