# Claude Code 設定

## IMPORTANT: エージェントルール

rules/agents.md のエージェント選択ルールに必ず従うこと。
コードレビュー依頼やコード変更後は**必ず** code-reviewer エージェントを実行すること。

サブエージェント（nuxt4-coder, typescript-coder 等）にタスクを委譲する際、**エージェントが専門知識として持っている判断領域にはオーケストレーターが踏み込まないこと。**

- ファイルの配置場所、コーディングパターン、プロジェクト構造に関する具体的なパスやディレクトリ名を**プロンプトで指定しない**
- 「何をしたいか（目的・要件）」を伝え、「どこに・どう実装するか」はエージェントに委ねる
- アーキテクチャレベルの判断（例: `shared/` vs `server/` の選択）が必要な場合のみ、オーケストレーターが方針を示す
- プランモードで実装計画を策定する際も同様に、具体的なファイルパスやディレクトリを計画に含めない。「スキーマを共通ファイルに切り出す」のように目的だけを記述し、配置場所の決定はエージェントに委ねる

## IMPORTANT: ナレッジ管理 (Obsidian CLI)

rules/obsidian-knowledge.md のナレッジ管理ルールに必ず従うこと。
タスク開始時の検索、メモの記録タイミング、タスク完了前のセルフチェックはすべて必須である。

## IMPORTANT: 課題作成テンプレート

Backlog 課題や GitHub Issue を作成する際は、rules/issue-template.md のテンプレートに必ず従うこと。

### コメント

- 関数を作成した場合はどのような処理を行う関数なのか簡潔にJSDOC形式で記載すること
- 見てわかるレベルの実装内容へのコメントは任意（不要なら書かない）
- 複雑な処理についてのコメント必須

### テストケース

- テストケースは `it` ではなく `test` を使って記述すること

### TypeScript

- 変数宣言は必ず `const` を用いること。`let` は使ってはならない
- `let` を使わないとどうしても実現できない場合のみ、ユーザーに確認を行ってから利用すること

## 検証

作業完了時、テスト実行・スクリーンショット・検証コマンドなど検証方法を必ず提供すること。
